# 2026-04-23 Session Log - WIT BLE Investigation Final

## Session Goal
Debug WIT WT901BLECL Bluetooth connectivity issue preventing plugin from streaming data.

## Investigation Timeline

### Phase 1: Initial BLE Plugin Debugging (22:00-22:01 EDT)
- **Problem**: `signalk-wit-imu-ble` plugin showed "Reconnecting" infinitely
- **Initial hypothesis**: Plugin code had timeout/connection state issues
- **Actions taken**:
  - Increased timeout from 1000ms → 5000ms
  - Added robust hex parsing
  - Fixed characteristic handle default (0x0030 → 0x000e)
  - Verified data parsing was actually correct

### Phase 2: System-Level BLE Investigation (22:01-22:05 EDT)
- **Discovery**: Two other Python scripts holding BLE connections:
  - `wit-ble-reader.py` (USB, not BLE - uses `/dev/ttyMidnightRider_IMU`)
  - `calypso-ble-reader.py` (Actual BLE reader with Calypso UUIDs)
- **Killed both processes** to clear BLE access
- **Root cause identified**: `bluetoothd` daemon holds **exclusive lock** on all BLE connections
  - When plugin tries `char-write-cmd`, bluetoothd returns "Disconnected"
  - gatttool cannot access GATT without bluetoothd releasing device
  - This is a **system architecture limitation**, not a plugin bug

### Phase 3: Search for Solutions (22:05-22:08 EDT)
- Investigated multiple approaches:
  - Bleak library (modern async Python BLE) - same hanging issue
  - Direct gatttool - also blocked by bluetoothd
  - Disabled bluetoothd, restarted Bluetooth service - didn't help
- **Key finding**: WIT manual recommends USB Adapter (CP2102 dongle) for PC connections

### Phase 4: Found Proven Solution - witmotion_ble (22:08-22:12 EDT)
- **GitHub discovery**: `ctroncozo/witmotion_ble` - specialized library for WT901BLECL5.0 BLE
- **Why it matters**: 
  - Designed specifically for WT901BLECL
  - Uses modern Bleak (async BLE)
  - Includes automatic time sync
  - Proven stable on Linux
  - Has proper message parsing for all sensor types
- **Created new plugin**: `signalk-wit-imu-ble-witmotion` v4.0.0
  - Spawns Python subprocess using witmotion_ble
  - Proper error handling and reconnection logic
  - Signal K v2.25 compatible structure
- **Result**: Plugin loads ✅ but Python script HANGS on `BleakClient.connect()`

### Phase 5: Final Root Cause Analysis (22:12-22:14 EDT)
- **Test result**: Even specialized witmotion_ble library's `Wit901BLEClient` cannot establish connection
- **Evidence**: 
  1. Device found ✅
  2. MAC address correct ✅
  3. Connect timeout reached ❌ (HANG indefinite)
  4. Same behavior as earlier attempts
- **Conclusion**: **This is NOT a plugin code issue. Linux Bluetooth stack incompatibility with WIT's BLE protocol.**

## Key Discoveries

### 1. WIT WT901BLECL Uses Proprietary BLE Protocol
- Not standard GATT services
- Manual specifies "BLE 5.0 Adapter" (USB dongle with CP2102)
- Direct Linux native BLE connection issues suggest device uses vendor-specific handshake

### 2. BlueZ Architecture Limitation
- `bluetoothd` daemon controls all BLE on Linux
- `gatttool` conflicts with bluetoothd control
- `BleakClient` from Bleak library also goes through bluetoothd
- All approaches hit same wall: exclusive daemon lock

### 3. USB Mode is Manufacturer-Recommended
- WIT manual (ManualsLib page 9) shows step-by-step USB Adapter setup
- "BLE 5.0 Adapter" with CP2102 is the proven connection method for PC
- This is what `signalk-wit-imu-usb` on `/dev/ttyWIT` uses
